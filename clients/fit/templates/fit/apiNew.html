Auth Service API

All routes are prefixed with /auth. The service runs on http://127.0.0.1:3297 when started locally.

Signup Flow

1️⃣ GET /auth/signup/first

Generate a per‑session salt for client‑side password hashing.

Request 
GET /auth/signup/first HTTP/1.1
Host: 127.0.0.1:3297
Cookie: session=<session‑id>
Response (200) 
{
    "salt": "<hex‑encoded salt>",
    "sessionId": "<session cookie value>"
}
Possible Errors
  • 500 – Internal server error (message: "Signup first step failed")

2️⃣ POST /auth/signup/second

Create the user using the previously‑generated salt and return an auth token.

Request 
POST /auth/signup/second HTTP/1.1
Host: 127.0.0.1:3297
Content-Type: application/json
Cookie: session=<session‑id>

{
    "passwordHash": "<client‑hashed password>",
    "username": "alice",
    "email": "alice@example.com"   // optional
}
Response (201) 
{
    "authToken": ["<authHead>", "<authTail>"]
}
Possible Errors
  • 400 – Missing required fields
  • 409 – User creation failed (e.g., duplicate username)
  • 500 – Unexpected server error

Login Flow

1️⃣ POST /auth/login/first

Retrieve the stored salt and a set of challenges for the supplied user.

Request 
POST /auth/login/first HTTP/1.1
Host: 127.0.0.1:3297
Content-Type: application/json

{
    "username": "alice"          // or "email": "alice@example.com"
}
Response (200) 
{
    "salt": "<hex‑salt>",
    "challenges": "<challenge‑data>",
    "sessionId": "<session‑id>",
    "servTime": 1734768392.123
}
Possible Errors
  • 400 – Username or email missing
  • 404 – User not found
  • 500 – Internal error

2️⃣ POST /auth/login/second

Submit the solved challenges to obtain an auth token.

Request 
POST /auth/login/second HTTP/1.1
Host: 127.0.0.1:3297
Content-Type: application/json
Cookie: session=<session‑id>

{
    "timestamps": [1629384000, 1629384010],
    "results":   ["<hash1>", "<hash2>"]
}
Response (200) 
{
    "authToken": ["<authHead>", "<authTail>"]
}
Possible Errors
  • 400 – Missing timestamps or results
  • 401 – Authentication failed
  • 500 – Unexpected error

Token Management

POST /auth/token/get

Validate an existing indexServer token and spawn a token for another service.

Request 
POST /auth/token/get HTTP/1.1
Host: 127.0.0.1:3297
Content-Type: application/json

{
    "authHead": "<head>",
    "timestamps": "1629384000",
    "challenges": "<challenge‑blob>",
    "results": "<result‑blob>",
    "targetServer": "myService",
    "expiry": 3600               // optional, seconds
}
Response (200) 
{
    "authToken": ["<newHead>", "<newTail>"]
}
Possible Errors
  • 400 – Missing required fields
  • 401 – Token verification failed
  • 500 – Server error

POST /auth/token/verify

Single‑step verification of any auth token.

Request 
POST /auth/token/verify HTTP/1.1
Host: 127.0.0.1:3297
Content-Type: application/json

{
    "authHead": "<head>",
    "sourceServer": "indexServer",
    "timestamps": [1629384000],
    "challenges": "<challenge‑blob>",
    "results": ["<result‑blob>"],
    "timeRecieveds": 1629384012   // optional
}
Response (200) 
{
    "verified": true,
    "result": "<authTail>"
}
Possible Errors
  • 400 – Missing fields
  • 401 – Verification failed
  • 500 – Server error

Client‑Side Hashing

The client never sends plain passwords. It uses the helper script below to produce a hash that combines three pieces of data:

#!/usr/bin/python
from authServer import hashCompute
from time import time

if __name__ == "__main__":
    print("Input in order: LocalHash/AuthHead(,Tail),timestamp,challenge ")
    print("Current time :", time())
    key = input("1 >")
    timestamp = input("2 >")
    challenge = input("3 >")

    print(hashCompute(key, timestamp, challenge))

What hashCompute does

  • Inputs
    • key – the local hash (for signup) or the authHead (for login/token verification).
    • timestamp – the Unix epoch time when the computation is performed (as a string or number).
    • challenge – the challenge string returned by /auth/login/first or /auth/token/get.
  • Algorithm – The function concatenates the three values and applies a cryptographic hash (currently SHA‑256) to the resulting bytes: 
    hash = SHA256( f"{key}{timestamp}{challenge}".encode('utf‑8') ).hexdigest()
    The output is a 64‑character hexadecimal string used as passwordHash (signup) or as an element of the results array (login / token verification).
  • Why it matters – By mixing the server‑issued salt/challenge with a time‑varying value, replay attacks are prevented and the server can verify the hash without ever seeing the raw password.

Health & Session Management

GET /auth/health

Liveness probe.

{
    "status": "healthy",
    "service": "AuthService"
}

POST /auth/session/clear

Clear any stored signup/login state and wipe the Flask session.

{
    "message": "Session cleared"
}
